# 2026.03.25 학부연구생 과제

## 1. Problem Definition

이번 과제의 핵심은 **raw data 수집 확대 + 키워드 품질 개선 + 크롤링 구조 고도화 방향 검토**이다.

기존 캐모마일 중심 수집에서 확장하여:

```text
티백
다크초콜릿
스트레스볼
```

3개 제품군의 **raw data(블로그 후기 데이터)** 를 수집해야 한다.

추가적으로, **키워드 설정의 주관성 문제**를 해결하고, 교수님이 언급한 **Claude 기반 크롤링/필터링 구조가 정확히 어떤 의미인지 정리**할 필요가 있다.

# 3. 교수님께 질문할 내용

## 질문 1. Claude를 써서 크롤링하라는 의미인가?

현재 해석한 구조는 다음과 같다.

```java
[1단계] 네이버 블로그 URL 수집
        → 네이버 검색 API

[2단계] 블로그 본문 크롤링
        → Playwright (JS 렌더링)

[3단계] 이상한 블로그 필터링
        → Claude API로 광고성/무관한 글 판별

[4단계] 감정/효과 분석
        → Ollama (로컬 LLM)

[5단계] 데이터 저장
        → DB (PostgreSQL 등)

[6단계] 키트 생성
        → 사용자 프롬프트 입력
        → RAG 검색
        → 키트 추천 출력
```

---

## 현재 의문점

### 1) Claude를 크롤링 자체에 쓰라는 의미인가?

가능성이 낮다.

Claude는 **웹 크롤러가 아니라 LLM** 이다.

따라서:

```text
크롤링 자체
❌ Claude

광고성/무관 글 판별
⭕ Claude
```

일 가능성이 높다.

즉:

```text
본문 크롤링
↓
Claude에게 판별 요청

"이 글이 실제 사용자 후기인지,
광고성 글인지,
무관한 글인지 판단해줘"
```

이런 구조일 가능성이 높다.

---

### 2) 비용 문제

Claude API 사용 시 비용이 발생한다.

문제점:

```text
데이터 수천 개 처리 시 비용 증가
속도 느려짐
API Rate Limit 가능성
```

예시:

```text
블로그 5000개
↓
Claude API 요청 5000번
↓
비용 증가
```

따라서 실무적으로는:

```text
Regex
↓
룰 기반 필터
↓
Claude는 애매한 글만
```

이 더 합리적이다.

---

## 교수님께 물어볼 질문 정리

```text
1. Claude를 "크롤링"에 쓰라는 의미인지,
   아니면 광고성/무관 블로그 판별용인지 궁금합니다.

2. Claude API를 사용하면 비용과 속도 문제가 있는데,
   전체 데이터에 적용하는 것인지,
   일부 애매한 데이터만 검증하는 방식인지 궁금합니다.

3. 현재 Regex + Rule-based filtering 이후
   Ollama 분류 구조를 유지해도 되는지 궁금합니다.
```

# 4. 키워드 설정의 핵심 고민

## 문제 정의

### 기존 방식의 문제

현재는 사용자가 직접 키워드를 설계하고 있다.

예:

```text
캐모마일 후기
캐모마일 불안
캐모마일 효과
캐모마일 차 마시고
```

문제:

```text
연구자의 주관이 들어감
편향 발생 가능
특정 방향 후기만 수집될 위험
```

예:

```text
"캐모마일 불안 완화"

→ 긍정 후기만 모일 가능성
→ 부정 후기 누락
```

교수님이 지적한 부분도 여기에 해당한다.

# 5. 해결 방향

## 핵심 아이디어

### "내가 생각한 키워드"가 아니라

### "유저가 실제 쓰는 말"을 사용한다.

즉,

```text
데이터 기반 키워드 확장
```

전략이다.

---

## 실무에서 가장 많이 쓰는 방법

### 기존 수집 데이터에서 역으로 키워드 추출

이미 수집한 CSV에서:

```text
title
description
```

컬럼을 분석해서

### 사람들이 실제 많이 쓰는 단어

를 뽑는다.

---

## 전체 흐름

```text
1차 데이터 수집
↓
title + description 분석
↓
형태소 분석
↓
빈도 높은 명사 추출
↓
신규 키워드 후보 생성
↓
2차 수집
```

즉,

```text
수집
→ 분석
→ 키워드 확장
→ 재수집
```

구조이다.

---

## 예시 코드

```python
from konlpy.tag import Okt

okt = Okt()

# title + description 합치기
all_text = " ".join(df["title"] + " " + df["description"])

# 명사 추출
nouns = okt.nouns(all_text)

# 빈도 높은 단어 찾기
```

---

## 기대 효과

예를 들어 캐모마일 후기에서:

```text
숙면
불안
긴장
허브티
릴렉스
카페인
진정
```

같은 단어가 많이 나오면

새로운 키워드 후보가 된다.

예:

```text
캐모마일 숙면
캐모마일 긴장
캐모마일 진정
캐모마일 카페인
```

---

## 장점

```text
주관성 감소
실제 사용자 표현 반영
키워드 편향 완화
재현 가능성 증가
```

즉, 연구 방법론 측면에서도 더 설득력 있다.

# 10. 현재 연구 방향 요약

```text
1. raw data 확대
   (티백, 다크초콜릿, 스트레스볼)

2. 키워드 편향 최소화
   → 형태소 분석 기반 확장

3. Claude 사용 목적 확인
   → 크롤링인지 필터링인지

4. 비용/속도 문제 검토
   → Regex → Rule-based → Claude 순 고려

5. 키트 구성 설계

6. 감정 기반 추천 시스템 고도화
   → RAG + 임베딩 구조
```

## Limitations

```text
1. Claude 활용 목적이 아직 명확하지 않다.
2. 키워드 자동 확장 기준(빈도 threshold)이 정해지지 않았다.
3. 형태소 분석 결과에는 노이즈 단어가 섞일 수 있다.
4. raw data 수집 규모에 따라 API 비용/시간 이슈가 발생할 수 있다.
5. 제품군별 키워드 전략이 달라질 가능성이 있다.
```
