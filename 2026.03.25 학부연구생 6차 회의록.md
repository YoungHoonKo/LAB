# 2026.03.25 학부연구생 회의 및 정리

## 1. Problem Definition

이번 회의의 핵심은 기존의 단순 LLM 감정분류 방식에서 벗어나, **임베딩, RAG, 차원 축소, 감정 점수화**를 적용하여 더 정교한 감성지수 기반 추천 시스템으로 확장하는 것이다.

현재까지는 블로그 본문을 수집한 뒤 LLM에게 감정분류를 요청하는 방식이었다. 앞으로는 수집한 텍스트를 임베딩하고, 벡터 DB에 저장한 뒤, 사용자 입력과 가장 유사한 데이터를 검색하여 제품을 추천하는 구조로 발전시킨다.

---

## 2. Core Concept

### 2.1 전체 연구 방향

최종 목표는 다음과 같다.

```text
재난·질병·심리적 위기 상황의 사용자
↓
감정 상태 입력
↓
사용자 입력을 임베딩 벡터로 변환
↓
기존 블로그 후기 데이터와 유사도 계산
↓
관련 제품 후기 검색
↓
LLM이 감성지수와 추천 근거 생성
↓
맞춤형 케어 키트 추천
```

즉, 기존 방식은 LLM에게 바로 추천을 맡기는 구조였지만, 앞으로는 **데이터 기반 검색 + LLM 추천** 구조로 바꾸는 것이다.

## 3. 기존 방식과 임베딩 추가 방식 비교

### 기존 방식

```text
블로그 텍스트 수집
↓
LLM 프롬프트로 감정분석 요청
↓
긍정/부정 결과 저장
↓
사용자 입력
↓
LLM이 매칭해서 제품 추천
```

### 기존 방식의 한계

```text
제품 수가 많아지면 LLM이 전체 데이터를 기억하지 못함
속도가 느려짐
추천 근거가 약해질 수 있음
유사한 후기 기반 검색이 어려움
```

---

### 임베딩 추가 방식

```text
블로그 텍스트 수집
↓
임베딩 변환
↓
벡터 DB에 저장
↓
사용자 입력도 임베딩 변환
↓
유사도 계산
↓
가장 비슷한 후기 검색
↓
제품 추천
```

### 장점

```text
수천 개 제품도 빠르게 검색 가능
사용자 입력과 의미가 비슷한 후기를 찾을 수 있음
LLM이 모든 데이터를 기억하지 않아도 됨
추천 속도와 정확도를 높일 수 있음
```

## 4. 전체 아키텍처

### 사전 처리 단계

```text
키워드
↓
URL 수집
↓
본문 크롤링
↓
청킹
↓
임베딩(BERT 또는 SBERT)
↓
NMF 또는 차원 축소

```

### 실시간 처리 단계

```text

↓

↓

```

정리하면 다음 구조이다.

```python
[사전 처리]

[실시간 처리]
고객 입력 → 임베딩 → DB 검색 → 관련 chunk → Ollama → 상품 추천
```

## 5. 임베딩(Embedding)

### 5.1 임베딩이란?

임베딩은 텍스트를 숫자 벡터로 변환하는 과정이다.

예시:

```text
"불안해서 잠이 안 와요"
↓
[0.21, 0.78, 0.13, 0.55, ...]
```

즉, 컴퓨터가 문장의 의미를 계산할 수 있도록 숫자 형태로 바꾸는 것이다.

---

### 5.2 “임베딩 뭐 썼어?”의 의미

누군가 “임베딩 뭐 썼어?”라고 물으면 다음 중 어떤 방법으로 텍스트를 숫자로 바꿨는지 묻는 것이다.

```text
TF-IDF
Word2Vec
BERT
SBERT
OpenAI Embedding
KoBERT
bge-m3
```

특징:

```text

```

이 프로젝트에서 사용할 임베딩은 3세대 문장/문맥 임베딩이 적절하다.

## 7. 임베딩과 딥러닝의 관계

### 핵심 정리

```text
임베딩은 딥러닝의 재료이고,
딥러닝은 임베딩을 만드는 공장이다.
```

### 감정분류 파이프라인

```text
텍스트 입력
↓
임베딩
↓
숫자 벡터
↓
분류기
↓
감정 레이블
```

예시:

```text
"BERT로 임베딩했다"
=
BERT 모델로 텍스트를 숫자 벡터로 바꾼 뒤 분류에 사용했다.
```

주의할 점:

```text
LLM에게 프롬프트로 "이 글 감정분류해줘"라고 요청한 것은 임베딩 방식이 아니다.
그것은 LLM 추론 기반 분류이다.
```

## 8. TF-IDF → NMF

### 8.1 TF-IDF란?

TF-IDF는 텍스트를 숫자 행렬로 바꾸는 고전적인 방법이다.

예시:

```text
"나는 사과를 좋아해"
↓
사과: 0.8
좋아: 0.6
나는: 0.1
```

즉, 단어가 얼마나 중요한지 숫자로 표현한다.

---

### 8.2 TF-IDF와 임베딩의 차이

| 구분      | TF-IDF         | 임베딩                |
| --------- | -------------- | --------------------- |
| 방식      | 단어 빈도 기반 | 의미 기반             |
| 문맥 이해 | 약함           | 강함                  |
| 예시      | 단어 출현 횟수 | 문장 의미 벡터        |
| 장점      | 빠르고 단순함  | 의미 유사도 계산 가능 |
| 단점      | 의미 이해 부족 | 계산량이 큼           |

비유:

```text
TF-IDF = 단어 출석부
임베딩 = 단어와 문장의 의미 지도
```

---

### 8.3 NMF란?

NMF는 Non-negative Matrix Factorization의 약자이다.

큰 행렬을 작은 행렬 두 개로 분해하는 방법이다.

```text
큰 행렬 V
↓
W × H
```

예시:

```text
문서 100개
↓
TF-IDF 행렬 생성
↓
NMF로 주제 패턴 분해
↓
정치 / 경제 / 스포츠 같은 주제 발견
```

---

### 8.4 TF-IDF → NMF 흐름

```python
문서들(텍스트)
    ↓
TF-IDF로 숫자 행렬 변환
    ↓
큰 행렬 V 생성
    ↓
NMF로 분해
    ↓
W × H
```

의미:

```text
W = 문서별 주제 가중치
H = 주제별 단어 패턴
```

## 9. 차원의 저주와 병목

### 9.1 차원의 저주란?

차원의 저주는 데이터의 특징 수가 너무 많아지면 계산이 느려지고 성능이 떨어지는 현상이다.

예시:

```text
단어가 100,000개
↓
TF-IDF 행렬이 100,000차원
↓
계산 느려짐
↓
메모리 부족
↓
성능 저하
```

---

### 9.2 Dimensionality와 Matrix 차이

| 용어      | 의미                              |
| --------- | --------------------------------- |
| Dimension | 데이터의 특징 개수                |
| Matrix    | 데이터를 행과 열로 표현한 숫자 표 |

예시:

```text
문서 100개 × 단어 10,000개
=
100 × 10,000 행렬
```

여기서:

```text
차원 = 10,000
행렬 = 100 × 10,000 형태의 숫자 표
```

---

### 9.3 NMF로 차원 축소

```text
100,000차원 큰 행렬
↓
NMF
↓
100차원 정도의 작은 주제 벡터
```

효과:

```text
계산 속도 향상
메모리 절약
주제 구조 파악 가능
```

## 10. LLM과 차원의 저주

질문:

```text
긴 블로그 본문을 그냥 LLM에게 넘기면 차원의 저주에 걸리는가?
```

답:

```text
정확히는 차원의 저주가 아니라 컨텍스트 윈도우 한계 문제가 생긴다.
```

### 차원의 저주가 생기는 경우

```text
TF-IDF처럼 텍스트를 직접 고차원 숫자 행렬로 바꿀 때
```

### LLM에서 생기는 문제

```text
한 번에 읽을 수 있는 텍스트 길이 제한
```

이것을 컨텍스트 윈도우라고 한다.

## 11. 긴 텍스트 처리 방법

### 방법 1. 청킹(Chunking)

긴 글을 작은 조각으로 나눈다.

```text
긴 블로그 글
↓
chunk 1
chunk 2
chunk 3
↓
각각 분석
↓
결과 통합
```

---

### 방법 2. 요약(Summarization)

긴 텍스트를 먼저 요약한 뒤 분석한다.

```text
긴 텍스트
↓
LLM 요약
↓
짧아진 텍스트
↓
감정분석
```

---

### 방법 3. RAG

RAG는 Retrieval Augmented Generation의 약자이다.

```text
검색 + 생성
```

전체 흐름:

```text
블로그 글 크롤링
↓
임베딩으로 숫자 벡터 변환
↓
벡터 DB 저장
↓
사용자 질문 입력
↓
질문도 임베딩 변환
↓
가장 비슷한 내용 검색
↓
관련 내용만 LLM에 전달
↓
LLM이 답변 생성
```

---

## 12. RAG 구조

RAG는 다음 구성요소로 이루어진다.

```text
RAG 시스템
├── 임베딩: 텍스트를 숫자로 변환
├── 검색: 관련 문서 찾기
└── LLM: 답변 생성
```

이 프로젝트에서는 RAG를 다음과 같이 사용할 수 있다.

```text
사용자 입력:
"요즘 불안하고 잠이 안 와요"

↓ 임베딩

벡터 DB에서 유사한 후기 검색

↓ 검색 결과

"캐모마일을 마시고 잠이 잘 왔다"
"라벤더 향을 맡고 안정감을 느꼈다"

↓ LLM

불안/불면 완화 키트 추천
```

## 13. KCI

KCI는 문맥에 따라 두 가지 의미로 사용될 수 있다.

### 13.1 Knowledge Conflict Injection

LLM이 원래 알고 있는 지식과 외부에서 주어진 컨텍스트가 충돌할 때 모델이 무엇을 믿는지 실험하는 개념이다.

예시:

```text
모델의 기존 지식:
파리는 프랑스의 수도이다.

주입된 컨텍스트:
파리는 독일의 수도이다.

질문:
파리는 어느 나라의 수도인가?
```

이때 모델이 기존 지식을 따르는지, 주어진 컨텍스트를 따르는지 확인한다.

---

### 13.2 Knowledge-grounded Conversational Intelligence

외부 지식 DB, 문서, 위키 등을 연결하여 더 정확하고 근거 있는 대화를 생성하는 시스템이다.

예시:

```text
단순 챗봇:
학습 데이터 기반으로 답변

KCI 시스템:
최신 DB 또는 문서를 검색한 뒤 근거 기반 답변
```

이 의미는 RAG와 연결된다.

## 14. Sentiment Analysis

Sentiment Analysis는 텍스트에서 감정이나 의견의 극성을 자동으로 파악하는 NLP 기술이다.

### 기본 분류

```text
긍정
부정
중립
```

예시:

```text
긍정: 이 제품 정말 좋아요.
부정: 완전 별로였어요.
중립: 오늘 배송이 왔습니다.
```

---

### 세밀한 분류

```text
기쁨
분노
슬픔
불안
놀람
안도
실망
```

---

### Aspect-based Sentiment Analysis

속성별로 감정을 분석하는 방식이다.

예시:

```text
배터리는 별로지만 화면은 좋아요.
```

분석:

```text
배터리: 부정
화면: 긍정
```

이 프로젝트에서는 제품별 감성 효과를 분석할 때 유용하다.

## 15. 감성지수 0~1 점수화

감성지수를 연속적인 실수로 표현한다.

예시:

```json
{
  "anxiety_relief": 0.82,
  "sleep_support": 0.71,
  "stress_reduction": 0.64,
  "positive_mood": 0.43
}
```

### 의미

```text
0에 가까움 = 효과 약함
1에 가까움 = 효과 강함
```

예시:

```text
캐모마일 후기에서 "잠이 잘 왔다", "마음이 편해졌다"가 많이 등장
↓
sleep_support: 0.78
anxiety_relief: 0.69
```

---

## 16. 3차원 벡터 스페이스 감성지수

감성지수를 3차원 벡터로 표현할 수 있다.

예시:

```text
x축 = 안정감
y축 = 활력
z축 = 만족도
```

제품별 감성 벡터:

```text
캐모마일 = [0.82, 0.20, 0.71]
초콜릿 = [0.35, 0.76, 0.68]
라벤더 아로마 = [0.91, 0.18, 0.74]
```

사용자 입력도 같은 공간에 배치한다.

```text
사용자 상태 = [0.88, 0.10, 0.20]
```

가장 가까운 제품을 추천한다.

## 17. 추천 시스템 개선 순서

추천 적용 순서는 다음이 합리적이다.

```text
1. 청킹 + RAG 먼저 적용
2. 그래도 느리면 NMF로 차원 축소 추가
3. 감정분석 정확도를 높이려면 BERT 임베딩 사용
4. Few-shot보다 Instruction Tuning과 Prompt Caching 검토
5. BM25로 키워드 매칭 성능 보완
```

## 18. BM25

BM25는 검색 알고리즘이다.

사용자의 입력과 문서 사이의 키워드 관련도를 계산한다.

예시:

```text
사용자 입력:
불안하고 잠이 안 와요

검색 문서:
캐모마일을 마시고 잠이 잘 왔다.
```

BM25는 다음 단어들의 관련성을 본다.

```text
불안
잠
캐모마일
```

RAG에서 벡터 검색과 함께 사용하면 검색 성능을 보완할 수 있다.

## 19. 모델을 학습시킨다는 뜻

모델을 학습시킨다는 것은 모델 내부의 파라미터를 데이터에 맞게 조정하는 것이다.

### 프롬프트 기반 추론

```text
모델 파라미터 변경 없음
입력만 주고 답변 받음
```

예:

```text
이 문장을 긍정/부정으로 분류해줘.
```

---

### 모델 학습

```text
입력 데이터 + 정답 라벨 제공
↓
오차 계산
↓
파라미터 업데이트
```

예:

```text
입력: 이 제품 좋아요
정답: 긍정
```

이런 데이터를 많이 넣어 모델이 감정분류를 더 잘하도록 가중치를 조정한다.

---

### 파인튜닝

이미 학습된 모델을 특정 작업에 맞게 추가 학습시키는 것이다.

예:

```text
기본 BERT
↓
한국어 제품 후기 감정 라벨 데이터로 추가 학습
↓
제품 후기 감정분석 특화 모델
```

## 24. 가상환경과 가상머신 차이

| 구분 | 가상환경                   | 가상머신                    |
| ---- | -------------------------- | --------------------------- |
| 의미 | Python 패키지 격리 공간    | 컴퓨터 자체를 가상으로 만듦 |
| 크기 | 가벼움                     | 무거움                      |
| 목적 | 프로젝트별 라이브러리 관리 | 다른 OS 실행                |
| 예시 | venv, conda                | VirtualBox, GCP VM          |

### 가상환경이 필요한 이유

프로젝트마다 라이브러리 버전이 다르기 때문이다.

예:

```text
프로젝트 A: numpy 1.0
프로젝트 B: numpy 2.0
```

가상환경이 없으면 한 프로젝트의 라이브러리 업데이트가 다른 프로젝트를 망가뜨릴 수 있다.

머신러닝에서는 PyTorch, TensorFlow, Transformers 등의 버전 차이가 크기 때문에 가상환경이 중요하다.

## 27. Limitations

현재 정리된 방향에는 다음 한계가 있다.

```text
1. 임베딩 모델을 아직 확정하지 않았다.
2. BERT, SBERT, OpenAI Embedding 중 무엇을 쓸지 비교가 필요하다.
3. NMF는 TF-IDF 기반 주제 분해에는 적합하지만, 문맥 임베딩과 결합할 때 설계가 필요하다.
4. 감성지수 0~1 점수화 기준이 아직 정의되지 않았다.
5. RAG 검색 성능은 벡터 DB와 임베딩 품질에 크게 의존한다.
6. Google Cloud GPU 사용은 비용이 발생할 수 있다.
7. 키트 추천의 실제 효과는 사용자 설문이나 실험으로 검증해야 한다.
```
