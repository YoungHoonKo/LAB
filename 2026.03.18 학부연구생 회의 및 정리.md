# 2026/03/18 학부연구생 회의 및 정리

## 1. Problem Definition

이번 과제의 핵심은 기존 프로젝트의 병목이었던 `본문 크롤링`과 `Ollama 감정분류` 속도를 개선하고, 수집한 블로그 본문에서 감정·증상·브랜드·상품명을 추출하여 케어 키트 추천 시스템의 데이터 기반을 강화하는 것이다.

해야 할 과제는 다음 3가지이다.

1. Gemma, GPT-OSS, Qwen으로 감정분류를 수행하고 정확도 비교하기
2. 네이버 스마트스토어 기준으로 케어 키트 15개씩 구성하기
3. 크롤링한 블로그 본문에서 상품명 추출하기

---

## 2. Core Concept

본 프로젝트는 정신적 어려움을 겪는 사람이나 재난적 상황에 있는 사람을 위해 맞춤형 케어 키트를 추천하는 사이트를 만드는 것을 목표로 한다.

전체 흐름은 다음과 같다.

```text
네이버 블로그 후기 URL 수집
→ 본문 크롤링
→ 광고성·무관한 블로그 제거
→ Ollama 기반 LLM 분석
→ 상품의 감정 효과, 증상, 브랜드 추출
→ 데이터베이스 구축
→ 사용자가 감정 상태를 입력
→ 수집된 데이터를 기반으로 맞춤형 키트 추천
```

예를 들어 사용자가 다음과 같이 입력한다.

```text
요즘 너무 불안하고 잠이 잘 안 와요.
```

그러면 시스템은 수집된 후기 데이터를 바탕으로 불안·불면 완화에 자주 언급된 제품군을 찾아 키트를 추천한다.

## 3. Logical Analysis

### 3.1 현재 문제점

Step 3의 본문 크롤링과 Step 4의 Ollama 분석이 7~8시간 이상 걸렸다. 이는 두 단계 모두 시간이 오래 걸리는 구조이기 때문이다.

### Step 3이 느린 이유

Step 3은 네이버 블로그 본문을 실제로 가져오는 단계이다. 이 과정은 네트워크 I/O 작업이다.

```text
네트워크 I/O = 외부 서버에 요청을 보내고 응답을 기다리는 작업
```

네이버 블로그 URL을 하나씩 순차적으로 요청하면, 한 URL의 응답이 끝날 때까지 다음 URL을 요청하지 못하므로 시간이 오래 걸린다.

### Step 4가 느린 이유

Step 4는 Ollama 로컬 LLM에게 본문 텍스트를 넣고 감정·증상·브랜드를 분석하게 하는 단계이다.

```text
모델 추론 = AI 모델이 입력을 읽고 결과를 생성하는 과정
```

LLM 추론은 계산량이 크기 때문에 가장 느린 부분이다. 특히 로컬 Ollama 모델은 PC 사양의 영향을 많이 받는다.

---

## 4. Solution

### 4.1 속도 개선 핵심 전략

속도 개선의 핵심은 다음 3가지이다.

```text
1. 병렬화
2. 필터링 선행
3. 모델 경량화
```

### 병렬화

기존에는 URL을 하나씩 요청했다.

```text
URL 1 요청 → 응답 대기 → URL 2 요청 → 응답 대기
```

개선 방식은 여러 URL을 동시에 요청하는 것이다.

```text
URL 1, URL 2, URL 3 ... URL 15 동시 요청
```

이를 위해 `requests` 대신 `aiohttp + asyncio`를 사용한다.

### 필터링 선행

LLM에 모든 본문을 보내면 시간이 오래 걸린다. 따라서 LLM 분석 전에 광고성 글, 무관한 글, 너무 짧은 글, 화장품·팩·카페 후기 등을 먼저 제거한다.

```text
불필요한 데이터 제거
→ LLM에 들어가는 데이터 수 감소
→ 분석 시간 감소
```

### 모델 경량화

모델이 작을수록 추론 속도가 빠르다.

```text
작은 모델 = 파라미터 수가 적은 모델
```

파라미터는 모델이 가진 가중치이며, 모델의 지식량과 복잡도에 해당한다. 파라미터 수가 적으면 일반적으로 속도와 메모리 측면에서 유리하다.

경량화 방법은 다음과 같다.

```text
1. 작은 모델 사용
2. 양자화 모델 사용
3. 입력 텍스트 길이 줄이기
4. 프롬프트 최소화
```

양자화는 모델 내부 계산 숫자를 더 단순하게 표현하여 메모리 사용량과 계산량을 줄이는 방법이다.

## 7. Step 3 Python Code

```python
import nest_asyncio
nest_asyncio.apply()
import re
import asyncio
import aiohttp
import pandas as pd
import time
from bs4 import BeautifulSoup

INPUT_FILE  = "step2_캐모마일.csv"
OUTPUT_FILE = "step3_캐모마일.csv"
LOG_FILE    = "log_step3_캐모마일.csv"
PRODUCT     = "캐모마일"

MIN_CONTENT_LENGTH      = 50
MAX_RETRIES             = 5
REQUEST_TIMEOUT         = 15
SHORT_CONTENT_THRESHOLD = 300
MAX_CONCURRENCY         = 15

AD_REGEX = re.compile(
    r"협찬|체험단|제공받았습니다|제공 받았습니다|유료광고"
    r"|소정의 원고료|원고료를 받고|광고입니다|서포터즈"
    r"|앰배서더|무상으로 제공|무료로 제공"
)

EXCLUDE_REGEX = re.compile(
    r"마스크팩|스킨케어|세럼|에센스"
    r"|만드는 법|레시피|DIY"
    r"|디퓨저|향초|아로마"
)

TOPIC_REGEX = re.compile(r"캐모마일|카모마일|케모마일")
CAFE_REGEX = re.compile(r"스타벅스|커피빈|투썸|공차|카페베네")
MENU_REGEX = re.compile(r"라떼|프라푸치노|주문했|음료수")

FIRST_PERSON_REGEX = re.compile(
    r"마셨어요|마셔봤|마시고|마신 후|마셨더니|마셨는데"
    r"|마셔보니|마시니|마시는 중|마시고 있|마셔왔|마셔볼"
    r"|먹어봤|먹었는데|먹고|먹어보니|먹는 중|챙겨 ?먹"
    r"|저는|제가|저도|저한테|제 경우"
    r"|사봤|구매해서|구입해서|주문해서|우려서"
    r"|꾸준히 마|매일 마|한 달째|일주일째|계속 마"
    r"|요즘 마|요즘 캐모마일|빠졌는데|즐겨 마"
)

HTML_SELECTORS = [
    ("div", {"class": "se-main-container"}),
    ("div", {"id": "postViewArea"}),
    ("div", {"class": "post-view"}),
]

HEADERS = {
    "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"
}

def parse_content(html: str) -> str:
    soup = BeautifulSoup(html, "html.parser")
    for tag, attrs in HTML_SELECTORS:
        el = soup.find(tag, attrs)
        if el:
            return el.get_text(separator=" ").strip()
    return ""

async def fetch_blog_content(session: aiohttp.ClientSession, url: str) -> str:
    mobile_url = url.replace("blog.naver.com", "m.blog.naver.com")
    for attempt in range(1, MAX_RETRIES + 1):
        try:
            async with session.get(
                mobile_url,
                headers=HEADERS,
                timeout=aiohttp.ClientTimeout(total=REQUEST_TIMEOUT)
            ) as resp:
                html = await resp.text(errors="replace")
                text = parse_content(html)
                if text:
                    return text
                print(f"본문 비어있음, 재시도 {attempt}/{MAX_RETRIES}: {url[:50]}")
        except Exception as e:
            print(f"오류 ({attempt}/{MAX_RETRIES}): {e} | {url[:50]}")
        await asyncio.sleep(0.5 * attempt)
    return ""

def apply_filters(row: dict, content: str) -> tuple[dict | None, str]:
    if AD_REGEX.search(content):
        return None, "광고제거"
    if len(content) < MIN_CONTENT_LENGTH:
        return None, "짧은본문"
    if not TOPIC_REGEX.search(content):
        return None, "주제무관"
    if EXCLUDE_REGEX.search(content):
        return None, "무관주제"
    if CAFE_REGEX.search(content) and MENU_REGEX.search(content):
        return None, "카페맛후기"
    if len(content) >= SHORT_CONTENT_THRESHOLD:
        if not FIRST_PERSON_REGEX.search(content):
            return None, "정보성글"
    result = dict(row)
    result["content"] = content[:3000]
    return result, "통과"

async def process_row(
    sem: asyncio.Semaphore,
    session: aiohttp.ClientSession,
    idx: int,
    total: int,
    row: dict
) -> tuple[dict | None, str, dict | None]:
    async with sem:
        print(f"크롤링 {idx}/{total}: {str(row['url'])[:60]}")
        content = await fetch_blog_content(session, row["url"])
        if not content:
            removed = dict(row)
            removed["제거사유"] = "크롤링실패"
            removed["content"] = ""
            return None, "크롤링실패", removed
        result, reason = apply_filters(row, content)
        print(f"→ {reason}")
        if result is None:
            removed = dict(row)
            removed["제거사유"] = reason
            removed["content"] = content[:200]
            return None, reason, removed
        return result, reason, None

async def crawl_all_async(df: pd.DataFrame):
    stat = {
        "크롤링실패": 0,
        "광고제거": 0,
        "짧은본문": 0,
        "주제무관": 0,
        "무관주제": 0,
        "카페맛후기": 0,
        "정보성글": 0,
        "통과": 0
    }
    results = []
    removed = []
    total = len(df)
    rows = df.to_dict("records")
    concurrency = min(MAX_CONCURRENCY, total)
    sem = asyncio.Semaphore(concurrency)
    connector = aiohttp.TCPConnector(limit=concurrency, ttl_dns_cache=300)

    async with aiohttp.ClientSession(connector=connector) as session:
        tasks = [
            process_row(sem, session, idx, total, row)
            for idx, row in enumerate(rows, 1)
        ]
        for coro in asyncio.as_completed(tasks):
            result, reason, removed_row = await coro
            stat[reason] += 1
            if result:
                results.append(result)
            elif removed_row:
                removed.append(removed_row)

    return pd.DataFrame(results), pd.DataFrame(removed), stat

if __name__ == "__main__":
    print("=" * 50)
    print(f"STEP 3 (고속): {PRODUCT} 본문 크롤링 + 2차 필터")
    print(f"동시 요청 수: 최대 {MAX_CONCURRENCY} | 타임아웃: {REQUEST_TIMEOUT}s | 재시도: {MAX_RETRIES}회")
    print("=" * 50)

    df = pd.read_csv(INPUT_FILE, encoding="utf-8-sig")
    before = len(df)
    df = df.drop_duplicates(subset="url").reset_index(drop=True)

    if before != len(df):
        print(f"중복 URL {before - len(df)}개 제거")

    print(f"입력: {len(df)}개\n")
    t0 = time.time()
    df_crawled, df_removed, stat = asyncio.run(crawl_all_async(df))
    elapsed = time.time() - t0

    df_crawled.to_csv(OUTPUT_FILE, index=False, encoding="utf-8-sig")
    df_removed.to_csv(f"step3_제거목록_{PRODUCT}.csv", index=False, encoding="utf-8-sig")
    pd.DataFrame([stat]).to_csv(LOG_FILE, index=False, encoding="utf-8-sig")

    print(f"\n저장 완료: {OUTPUT_FILE} ({len(df_crawled)}개)")
    print(f"제거 목록: step3_제거목록_{PRODUCT}.csv ({len(df_removed)}개)")
    print(f"로그 저장: {LOG_FILE}")
    print(f"소요 시간: {elapsed:.1f}초 ({elapsed/60:.1f}분)")
    print("\n[필터링 통계]")
    for k, v in stat.items():
        print(f"  {k}: {v}개")
```

## 11. Regex

Regex는 특정 패턴의 글자가 있는지 찾는 도구이다.

```python
import re

text = "이 제품은 협찬을 받았습니다"

if "협찬" in text:
    print("광고")

AD_REGEX = re.compile(r"협찬|체험단|원고료")

if AD_REGEX.search(text):
    print("광고")
```

`|`는 “또는”이라는 뜻이다.

```text
협찬|체험단|원고료
=
협찬 또는 체험단 또는 원고료
```

`re.compile()`은 정규표현식 패턴을 한 번만 해석해 저장해두는 것이다. 데이터가 많을 때는 반복문 안에서 `re.search()`를 직접 쓰는 것보다 효율적이다.

나쁜 방식:

```python
for row in rows:
    if re.search(r"협찬|체험단|원고료", content):
        ...
```

좋은 방식:

```python
AD_REGEX = re.compile(r"협찬|체험단|원고료")

for row in rows:
    if AD_REGEX.search(content):
        ...
```

`.search(text)`는 해당 텍스트 안에 정규표현식 패턴이 존재하는지 찾는 함수이다.

## 12. 비동기 방식

### 동기 방식

```python
for url in urls:
    content = requests.get(url)
```

동기 방식은 하나가 끝나야 다음 작업을 시작한다.

```text
URL 1 요청 → 대기 → 완료
URL 2 요청 → 대기 → 완료
URL 3 요청 → 대기 → 완료
```

### 비동기 방식

```python
async def fetch(url):
    async with session.get(url) as resp:
        return await resp.text()
```

비동기 방식은 여러 요청을 동시에 보낸다.

```text
URL 1 요청
URL 2 요청
URL 3 요청
...
응답 오는 순서대로 처리
```

## 13. async / await / asyncio

### async

`async`는 이 함수 안에 기다리는 작업이 있음을 표시한다.

```python
async def fetch_blog_content(session, url):
    ...
```

### await

`await`는 두 가지 의미를 가진다.

```text
1. 여기서 기다린다.
2. 기다리는 동안 다른 작업을 처리할 수 있게 한다.
```

예시:

```python
html = await resp.text()
```

### asyncio

`asyncio`는 async/await로 만든 비동기 함수들을 실행하는 관리자이다.

```python
asyncio.run(crawl_all_async(df))
```

비동기 함수는 그냥 호출하면 실행되지 않는다.

```python
main()
```

이 아니라 다음처럼 실행해야 한다.

```python
asyncio.run(main())
```

## 16. Step 4: Ollama 병렬 분석

### 16.1 Step 4의 목적

Step 4는 Step 3에서 통과한 본문을 Ollama에 입력하여 감정, 증상, 브랜드를 추출하는 단계이다.

기존 목표:

```text
감정분류
```

확장 목표:

```text
감정
증상
브랜드
상품명
효과
```

### 16.2 Ollama 병목 해결 방법

#### 1. 배치 처리

하나씩 보내지 않고 여러 개를 묶어서 보낸다.

```python
batch = texts[i:i+5]
```

요청 횟수가 줄어들어 속도가 개선될 수 있다.

#### 2. 작은 모델 사용

모델 크기가 작을수록 추론 시간이 짧아진다. 대신 성능이 낮아질 수 있으므로 정확도와 속도를 함께 비교해야 한다.

#### 3. 프롬프트 최소화

프롬프트가 길수록 토큰 수가 증가한다. 토큰 수가 많을수록 LLM 처리 시간이 길어진다. 따라서 지시문은 짧고 명확해야 한다.

#### 4. 병렬 LLM 호출

여러 요청을 동시에 보내 CPU/GPU 사용률을 높인다.

#### 5. 캐싱

이미 처리한 텍스트는 다시 처리하지 않는다.

```text
같은 content hash가 있으면 기존 결과 재사용
```

## 17. 상품명 추출

### 17.1 목적

크롤링한 본문에서 실제로 언급된 상품명과 브랜드를 추출한다.

예시:

```text
티젠 캐모마일
쌍계명차 캐모마일
오설록 릴렉싱 티
스타벅스 캐모마일 릴렉서
```

### 17.2 추출 방식

상품명 추출은 두 가지 방식으로 접근할 수 있다.

```text
1. Regex 기반 후보 추출
2. LLM 기반 브랜드·상품명 추출
```

Regex는 빠르지만 문맥 이해가 어렵다. LLM은 문맥 이해가 가능하지만 느리다. 따라서 Regex로 후보를 줄이고, LLM으로 정제하는 방식이 합리적이다.

## 18. 광고가 이미지로 나타나는 문제

일부 네이버 블로그는 광고 문구가 텍스트가 아니라 이미지 안에 들어 있다. 이 경우 Regex로는 감지할 수 없다. Regex는 텍스트에만 적용되기 때문이다.

### 가능한 해결 방법

#### 방법 1. 이미지 OCR

이미지를 다운로드한 뒤 OCR로 이미지 속 글자를 추출한다.

```text
본문 이미지 수집
→ OCR 적용
→ 이미지 속 텍스트 추출
→ 광고 문구 탐지
```

단점:

```text
속도가 느림
오류 가능성 있음
구현 복잡도 증가
```

#### 방법 2. 이미지 개수/링크 패턴 기반 휴리스틱

OCR 없이 광고 가능성을 추정한다.

예시 기준:

```text
이미지가 과도하게 많음
구매 링크가 많음
스마트스토어 링크 포함
제품명 반복
브랜드명 반복
```

#### 방법 3. 제거하지 않고 광고 의심 태그 부여

완전히 제거하기 어렵다면 `ad_suspected=True` 컬럼을 추가한다.

```text
확실한 광고 → 제거
의심 광고 → ad_suspected=True
일반 후기 → 통과
```

현실적인 1차 해결책은 다음이다.

```text
텍스트 광고는 Regex로 제거
이미지 광고는 ad_suspected=True로 태깅
필요 시 일부만 수작업 검토
```

## 22. Limitations

현재 방식의 한계는 다음과 같다.

```text
1. Regex 기반 광고 제거는 텍스트 광고만 탐지할 수 있다.
2. 이미지 광고는 OCR 없이는 정확히 탐지하기 어렵다.
3. 비동기 크롤링은 빠르지만 서버 차단 가능성을 고려해야 한다.
4. LLM 병렬 호출은 PC 자원 한계에 따라 실패할 수 있다.
5. 모델별 정확도 비교는 수작업 정답 라벨이 있어야 가능하다.
6. 상품명 추출은 브랜드명과 일반 명사를 구분하는 문제가 있다.
7. 스마트스토어 키트 구성은 가격·재고·배송 상황에 따라 바뀔 수 있다.
```
